Drive one intent from ICE to merge-ready through a fixed pipeline, accumulating a `idsd-ship-report.md` digest of what needs the human's attention. You **orchestrate** existing skills — invoke them, never reimplement; their rules still hold (gates absolute, follow-ups routed before archive, no-mocks, …).

**Interactive first.** Prefer asking the human live over deferring to the digest. A decision you can't settle from the intent, the code, or a sensible default — or an ambiguity that changes what gets built — is *blocking*: stop and ask now. The sub-skills' own clarify gates (e.g. `idsd-build`'s Phase 1) still fire — never suppress one by recording instead. `idsd-ship-report.md` is for what does *not* block — surfaced for the human at the final checkpoint, not lost in chat history.

Where this fits: `idsd-intent` → (`idsd-audit`) → **`idsd-ship`** (= `idsd-build` + `/review` + `/security-review` + `/refactor`, sequenced).

Input: an intent `NNN-<slug>` (as `idsd-build` takes). If unspecified, list the not-yet-built ones and ask which.

`idsd-ship-report.md` lives at the repo root, **overwritten each run** (latest build only; git history is the durable record). Add it to `.gitignore` if it isn't already. **Attention** is the *non-blocking* set: decisions the human can ratify at the checkpoint, deferrals, watches, gated/declined fixes — never a blocking question (asked live, per Interactive first) nor routine work already in code, the ICE, or the commit. Each item leads with the decision (or watch) and what the human must do, at their altitude — no "what we tried", no dep lists, command strings, or code identifiers; those live in the diff, commit, or ICE, so link rather than restate. One item, one line where it can. **The test: if the human takes no action, it is not attention; omit it.** There is no informational/FYI tier (no "here's what changed" or "no decision needed" entry — that's the diff), and no verification narration (what passed, what's byte-identical, a checklist of invariants confirmed). A stage that ran clean with no attention items gets `Passed — nothing needs your attention.` under its heading, standing alone — no parenthetical or why-clause; don't pad. A stage *skipped* for not applying uses `Not applicable — <reason>` instead.

## Pipeline

Start by resetting `idsd-ship-report.md` from `templates/ship-report-template.md`, stamping the intent it covers. Don't discard silently: if the current file holds items from a run that never reached merge, surface them for the human first — a merged run's items are already durable in the ICE and git. Then run the stages in order, recording each attention item into its stage's section the moment it surfaces, not batched at stage end.

**How each stage runs.** Build runs inline — it needs the human live (restate/confirm, clarify gates). Code-review, security-review, and refactor each run in a **dedicated subagent** that executes the skill in full and returns its findings as structured data — the subagent never decides whether to run, and never fakes the pass with its own inline judgment. Only the main thread has the human, so every decision and all report-writing stay here: take the subagent's returned findings, ask the human live for any that block, record the rest. When a subagent hits something only the human can settle — a clarification, a gated choice — it pauses and returns `blocked: <what it needs>` rather than guessing. Answer it, then **resume that same subagent by its ID** so it continues with its context and progress intact; never start a fresh one — a new spawn loses the work it already did and the skill state it was holding.

1. **Build** — run `idsd-build` for the intent in its **pipeline mode**: it runs restate/confirm, context, and implementation until gates are green, then hands back — skipping its self-review, checkpoint, and merge, which the dedicated passes and the final approval below replace. As it builds, idsd-build routes every follow-up to the ICE's `## Follow-ups` and every durable standard to a constitution proposal — its own rules, not deferred to merge. Before recording, confirm it did: an unrouted follow-up is a build defect, not something the report absorbs.
   - Record under **# Build**: ambiguities resolved and how, deferrals to confirm, constraints that can't become a gate (need human judgment), and decisions to ratify — each pointing to its durable home (the ICE `## Follow-ups`, a constitution proposal) that idsd-build already wrote. The report flags for the human; it never replaces the durable record.
2. **Code-review** — spawn a subagent to run `/review` on the build's changes: it applies every fix it can make correctly and returns the rest — findings needing a human decision (a trade-off, an ambiguous intent, a risky change), plus any behaviour-changing fix it made. On its return, ask the human live for blocking findings; record the others.
   - Record under **# Code-review**: findings left for the human (each with the decision needed), plus any behaviour-changing fix they should know about.
3. **Security-review** — *only if* the change touches a security surface (input handling, filesystem/network/exec, auth or session, secrets, deserialization, or any constitution security invariant); otherwise skip. Spawn a subagent to run `/security-review` on the build's changes: it applies trivial safe fixes (e.g. secret redaction) and returns the rest as findings with severity, exploit scenario, and fix. On its return, ask the human about anything blocking; record the others.
   - Record under **# Security**: findings left for the human (severity + exploit + fix) and any trivial fix applied — or `Not applicable — no security surface touched.`
4. **Refactor** (loop, max 3 iterations) — the pipeline's safety net for completeness; don't shortcut it. Each iteration spawns a **fresh** subagent (not a resume) to run `/refactor` on the changes: it applies routine fixes, returns gated architecture/testing/project-setup ones as proposals (not applied), and reports whether the change is now fully compliant. Spawning fresh re-reads the now-changed tree, catching fixes that only surface after earlier ones land. Stop the moment a subagent reports compliant; otherwise iterate to the cap. The blocked→resume rule still holds *within* an iteration — if that pass's subagent pauses, resume it by ID; only a new iteration spawns fresh. On each return, ask the human about gated proposals; record what's deferred or declined.
   - Record under **# Refactor**: gated fixes proposed or declined across iterations, watches deferred (e.g. duplication under the extract threshold), and — if it hit the cap without reaching compliant — that fact with what's still open.
5. **Tighten** (optional) — *only if* the build added or changed standalone prose (docs, markdown) or non-trivial comment blocks; otherwise skip. Spawn a subagent to run `/tighten` on those files. On its return, record what it flagged but couldn't fix.
   - Record under **# Tighten**: prose `/tighten` flagged but couldn't fix — or `Not applicable — no prose changed.`
6. **Summary** — write **# Summary**: 2–3 lines — what shipped and the single most important thing (if any) to resolve before merge. The decisions are already recorded in the stage sections above — point to them, don't re-list. At the human's altitude: no metrics (test counts, coverage), no file/path trace — gate evidence is presented separately at the checkpoint.

## After the pipeline

Present the `idsd-ship-report.md` file itself — its sections as written — alongside `idsd-build`'s checkpoint evidence (gate + scenario + observed outcomes); point the human at the file, don't re-author it as a parallel chat digest (per *Keep chat lean*). On the human's approval, hand back to `idsd-build`'s merge step — `status: built`, archive, roadmap, commit (which asks first). The pipeline never commits on its own.

## Rules

- Adds only sequencing and the digest; the sub-skills own the actual work.
- Keep chat lean — write attention items to `idsd-ship-report.md`, never into a chat summary, and don't echo the digest back in prose.
- A stage that hard-fails (red gate, build can't complete) stops the pipeline — surface it, don't push through to review/refactor.
- Never relax a sub-skill's gate to keep the pipeline moving; a blocking decision is asked live, not recorded to dodge it (Interactive first).
